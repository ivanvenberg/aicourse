## 5.1 Что такое API

**API** — это способ для программ общаться между собой.

**Аналогия:** Официант в ресторане. Ты (приложение) говоришь официанту (API) что хочешь. Официант идёт на кухню (сервис) и приносит результат. Ты не знаешь как устроена кухня — тебе нужен только официант.

**Когда ты пишешь Telegram-боту:**
```
Ты → пишешь сообщение
  → Telegram API принимает
  → передаёт твоей программе
  → программа отправляет запрос к AI (через OpenRouter)
  → AI отвечает
  → программа отправляет ответ обратно в Telegram
  → ты видишь ответ
```

Ты не видишь эту цепочку. Ты просто пишешь боту и получаешь ответ.

## 5.2 OpenRouter: один ключ ко всем моделям

Из «The Vibe Coding Bible»:

> *«Рынок AI моделей меняется каждую неделю. Лучшая стратегия — не привязываться к одному провайдеру. Используй агрегатор.»*

**OpenRouter** — сервис который даёт доступ к Claude, Gemini, GPT, Mistral и другим через один API.

**Почему это удобно:**
- Одна регистрация → все модели
- Легко переключаться: не устраивает Claude → попробуй Gemini → не устраивает → попробуй GPT
- Наценка всего +5% от цены провайдера

**Для бота поддержки — Gemini Flash:**
- В 5-10 раз дешевле чем Claude Opus
- Достаточно умный для ответов на типовые вопросы
- Быстрее отвечает

**Когда использовать какую модель:**

| Задача | Модель | Почему |
|--------|--------|--------|
| Бот поддержки, типовые вопросы | Gemini Flash | Дёшево и быстро |
| Написание постов в стиле автора | Claude Sonnet | Качественный текст |
| Сложный анализ, планирование | Claude Opus | Лучшее рассуждение |
| Простые скрипты, утилиты | Sonnet / Gemini | Экономия |

Регистрация: [openrouter.ai](https://openrouter.ai)

## 5.3 Как работает Telegram-бот

**Шаги:**
1. Регистрируешь бота через @BotFather → получаешь токен
2. Твой код запускается (локально или на сервере)
3. Бот слушает входящие сообщения (polling — спрашивает каждые несколько секунд «есть новые?»)
4. Получил сообщение → отправил в AI → получил ответ → отправил обратно

**Токен бота:** строка вида `123456:ABCdef...`
- Это уникальный ключ твоего бота
- Никому не показывай
- С ним можно управлять ботом

**Polling vs Webhook:**
- Polling: проще, для разработки
- Webhook: Telegram сам присылает сообщения, нужен для продакшена

## 5.4 RAG: научи бота отвечать по твоим данным

**RAG (Retrieval-Augmented Generation)** — метод когда AI отвечает не из своей головы, а на основе конкретных документов которые ты ему дал.

**Без RAG:**
> «Расскажи о политике возврата» → AI придумывает правдоподобный, но неверный ответ

**С RAG:**
> AI читает файл с реальной политикой возврата → отвечает точно по документу → если вопрос вне документа — честно говорит что не знает

**Как сделать:**
```
Добавь в бота базу знаний из файла knowledge.md.
Бот должен отвечать только на основе этого файла.
Если вопрос выходит за пределы — честно говорить что не знает.
Не придумывай информацию которой нет в файле.
```

**Примеры по профессиям:**

*Дизайнер:* бот знает твой прайс-лист, политику правок, типичные вопросы клиентов → отвечает на них за тебя

*Главред:* бот знает редполитику → проверяет тексты авторов

*Архитектор:* бот знает типовые условия по проектам → отвечает клиентам на базовые вопросы

*СМО:* бот знает метрики и планы → отвечает на вопросы команды

## 5.5 Multi-Agent Review: проверка кода несколькими агентами

Из «The Vibe Coding Bible»:

> *«Один агент строит, другой проверяет. Это как архитектор и инспектор: один проектирует, другой проверяет что всё по нормам.»*

**Multi-Agent Review** — паттерн где один агент создаёт код, другой его ревьюит.

**Как работает:**
```
Создай двух агентов:
1. Агент-Разработчик: создаёт функционал
2. Агент-Ревьюер: проверяет код и находит:
   - уязвимости безопасности
   - крайние случаи которые не обработаны
   - что сломается в продакшне

После ревью — Разработчик исправляет, Ревьюер проверяет снова.
```

**Зачем это нужно:**
- AI в режиме «создания» и AI в режиме «проверки» — это разные режимы
- Ревьюер находит то что пропустил разработчик
- Из книги «The Vibe Coding Bible»: *«Повар пробует перед тем как подать. Дегустация — другой режим чем готовка»*

**Промпт для самопроверки:**
```
Теперь проверь код который только что написал. Найди:
- что может сломаться в реальном использовании
- уязвимости безопасности
- что упустил
- что бы ты изменил если бы это шло в продакшн
```

## 5.6 Безопасность: обязательная программа

Один разработчик из книги не проверил безопасность — его сервер взломали и использовали для майнинга криптовалюты через несколько часов после деплоя.

**Обязательный промпт перед любым деплоем:**
```
Проверь безопасность этого приложения:
1. Нет ли API ключей прямо в коде (должны быть в .env)
2. Правильно ли настроены права доступа к базе данных
3. Есть ли защита от спама и злоупотреблений
4. Можно ли получить чужие данные без авторизации
Исправь всё что найдёшь. Объясни что сделал.
```

**Золотые правила:**
- API ключи → только в `.env` файле, никогда в коде
- `.env` файл → никогда в Git (добавь в `.gitignore`)
- Перед деплоем → запусти проверку безопасности

## 5.7 Автоматизация: Personal OS

Всеволод Устинов, основатель Plurio.AI:

> *«AI-native начинается не с инструментов. Начинается с единой точки входа в контекст — твой Personal OS. Сначала ты учишься сам работать с агентами. Потом это превращается в командные процессы. Нельзя делегировать AI-трансформацию «отделу AI» — если ты сам не можешь через агента улучшить свой процесс, переход не произошёл.»*

**Personal OS для каждой профессии:**

*Главред:* скилл `find-news` + скилл `write-posts` + бот который отвечает авторам на типовые вопросы = 3 часа рутины → 20 минут

*Дизайнер:* скилл `brief-generator` + бот который принимает правки от клиентов + MCP Figma = безграничный контроль над проектами

*СМО:* скилл `weekly-report` + скилл `competitor-watch` + бот для команды = аналитика которая работает сама

*CRM маркетолог:* скилл `segment` + скилл `write-emails` + автоматическая база = воронки без ручного труда

## 5.8 Vibe Wall: система документации

Из «The Vibe Coding Bible»:

> *«Vibe Wall — это место где ты фиксируешь всё что узнал о проекте: архитектурные решения, что пробовал и почему не сработало, промпты которые работают. Это корпоративная память которая не теряется.»*

**PROMPTS.md** — файл где ты сохраняешь лучшие промпты:
```markdown
# Промпты которые работают

## Для создания брифа
[промпт который дал лучший результат]

## Для анализа данных
[промпт для скрипта]

## Для ревью безопасности
[промпт который нашёл все проблемы]
```

Это твоя библиотека. Строится постепенно. Со временем становится ценнее самого кода.
