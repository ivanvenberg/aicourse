## 5.0 Домен и DNS (простыми словами)

**Домен** — человеческое имя (`moy-produkt.ru`).  
**DNS** — таблица «этот домен → вот IP/хостинг».

Цепочка:
```
Купил домен (Namecheap и т.п.)
  → в настройках DNS указал записи от Vercel
  → Vercel выдал сертификат HTTPS
  → ссылка открывается
```

Нет домена к демо — **нормально**. Важнее: продукт по ссылке + уверенный питч. Домен — плюс.

## 5.1 Git при вайбкодинге

Git — система контроля версий. Каждый `git commit` — точка сохранения. AI удалил файл → можно вернуться.

**Почему особенно важно при вайбкодинге:** AI может удалить код без предупреждения. Коммит — страховка.

### Семантические коммиты

```
feat: добавить авторизацию через email
fix: исправить ошибку при пустом поле
refactor: разбить функцию на модули
docs: обновить README
chore: обновить зависимости
```

Через месяц — история проекта читается как дневник.

**Промпт:**
```
После каждого изменения делай git commit с семантическим сообщением.
Формат: [feat/fix/refactor/docs/chore]: краткое описание что изменилось
```

### Git-стратегия

**Коммить перед каждой большой задачей AI** — точка возврата если что-то пойдёт не так.

**Ветки для экспериментов:**
```bash
git checkout -b experiment/new-ui
# вайбкодишь...
# понравилось → git merge
# нет → git branch -D experiment/new-ui
```

**Никогда не удаляй файлы** — в AGENTS.md добавь: «Никогда не удаляй файлы, перемещай в /archive/»

**Смотри diff перед принятием:**
```bash
git diff
```
Много удалённых строк — спроси AI почему.

## 5.2 Code Review кода которого ты не писал

Новый навык который критичен при вайбкодинге.

### Стратегия 1: Попроси объяснить

```
Объясни этот код как будто я не программист.
Что делает каждая функция? Почему именно так?
```

Если не может объяснить просто — код слишком сложный.

### Стратегия 2: Адвокат дьявола

```
Ты только что написал этот код. Теперь притворись
что ты злобный code reviewer. Найди всё что может быть неправильным.
```

### Стратегия 3: Чек-лист

```
Проверь по чек-листу:
□ Обрабатываются ли все ошибки?
□ Что будет при пустом вводе?
□ Нет ли жёстко заданных паролей или ключей?
□ Понятны ли имена переменных и функций?
□ Есть ли дублирование кода?
```

### Стратегия 4: Тест как документация

```
Напиши тесты для этого кода.
Тесты должны служить документацией:
читая их — должно быть понятно что делает каждая функция.
```

## 5.3 Технический долг

**Технический долг** — компромисс сделанный сейчас который придётся исправлять потом.

При вайбкодинге долг накапливается **в 10 раз быстрее** чем при ручном кодировании:
- AI генерирует быстро — берёт «кредит» у будущего
- AI оставляет мусор: debug-логи, `interim_result5`, закомментированный код
- Одна проблема решается созданием трёх новых

### Рефакторинг по расписанию

После каждой крупной фичи:
```
Посмотри на последние изменения. Найди:
- дублирующийся код
- функции длиннее 30 строк
- непонятные имена
- закомментированный код
- debug-логи
Упрости, не ломая функциональность.
```

### Когда рефакторить

✅ После крупной фичи / когда непонятно почему что-то сломалось / третья похожая функция

❌ На горящем дедлайне / если код работает и не трогают

## 5.4 Граница контроля: когда взять управление руками

### Когда остановиться

**Ты перестал понимать что происходит:**
```
Перестань генерировать. Попроси объяснить.
Если не понимаешь объяснение — упрости задачу.
```

**AI зациклился:**
Разные формулировки → та же ошибка → начни новый чат, опиши иначе, или сделай маленький кусок руками.

**Изменение ломает несвязанные вещи:**
Архитектурная проблема — нарисуй структуру на бумаге, реши в голове, потом объясни AI.

**Критическая безопасность:**
Авторизация, платежи, персональные данные — проверяй каждую строку самостоятельно.

### Как не потерять понимание кода

- Читай diff перед каждым принятием
- Проси объяснения всегда
- Пиши тесты которые показывают намерение
- Веди PROMPTS.md — сохраняй важные архитектурные решения

## 5.5 Будущее профессии: честный разговор

### Что автоматизируется

- Написание шаблонного кода (уже автоматизировано)
- Поиск и исправление понятных багов
- Рефакторинг по правилам
- Написание тестов для существующего кода
- Документация

### Что остаётся за человеком

- Понимание проблемы пользователя (AI не знает твоих пользователей)
- Архитектурные решения (trade-offs, «что если вырастем в 10 раз»)
- Суждение: увидеть что правдоподобный ответ неправильный
- Этика и ответственность (AI не несёт ответственности — ты несёшь)

### Авторство

Код написал AI → ты его принял → это твой код.
Бот ответил неправильно → твой бот → твоя ответственность.

Вайбкодинг не снимает ответственности. Он меняет инструмент, не роль.

### Как строить карьеру

**Не конкурируй с AI** в том что AI делает хорошо.

**Развивай:**
- Глубокое понимание своей области (ты знаешь дизайн, редактуру, маркетинг — AI нет)
- Суждение: видеть когда AI неправ
- Скорость обучения — инструменты меняются каждые месяцы

**Комбинируй профессии:**
Дизайнер который вайбкодит = дизайнер который может сделать прототип сам. Это другой уровень возможностей.

## 5.6 Демо-день: питч покупателю (5 минут)

Не «я прошёл курс». **«Вот за что заплатите».**

| Блок | Время | Что сказать |
|------|-------|-------------|
| Кому | 30 сек | Из `brief.md` — конкретный покупатель |
| Боль в деньгах | 30 сек | Сколько стоит без продукта |
| Живое демо | 2 мин | **Платная фича** — то, за что в brief цена |
| Цена | 30 сек | Гипотеза ₽/мес или за раз |
| Доказательство | 30 сек | Цитата «заплатил бы / возьму на работу» |
| Шаг | 30 сек | Один шаг на месяц: первый платящий, пилот, доработка |

Сломалось на демо — ок, чини вслух. Покупателю важнее честность, чем идеальный клик.

**Минимум к зачёту:** ссылка + `brief.md` + одна цитата от человека вне курса.
