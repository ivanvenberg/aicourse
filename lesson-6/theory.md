## 5.1 Git при вайбкодинге

Git — система контроля версий. Каждый `git commit` — точка сохранения. Если что-то сломалось — можно вернуться.

При вайбкодинге Git особенно важен. AI может удалить код без предупреждения. Коммит — страховка.

### Семантические коммиты

Коммит-сообщение описывает что именно изменилось:

```
feat: добавить авторизацию через email
fix: исправить ошибку при пустом поле ввода
refactor: разбить функцию на модули
docs: обновить README
chore: обновить зависимости
```

**Почему важно:** через месяц ты (или AI) сможешь понять историю проекта из коммитов.

**Промпт для AI:**
```
После каждого изменения делай git commit с семантическим сообщением.
Формат: [feat/fix/refactor/docs/chore]: краткое описание
```

### Git-стратегия для AI-проектов

**Правило 1: Коммить часто**
Перед каждой большой задачей AI → сделай коммит. Это точка к которой можно вернуться.

**Правило 2: Ветки для экспериментов**
Хочешь попробовать новый подход → создай ветку:
```
git checkout -b experiment/new-ui
# вайбкодишь...
# понравилось → git merge
# не понравилось → git branch -D experiment/new-ui
```

**Правило 3: Никогда не удаляй файлы**
В AGENTS.md добавь: «Никогда не удаляй файлы — перемещай в /archive/»

**Правило 4: Проверяй diff перед принятием**
Когда AI сделал изменения — посмотри что изменилось:
```
git diff
```

Смотри на количество удалённых строк. Если AI удалил много — спроси почему.

## 5.2 Code Review кода которого ты не писал

Это новый навык который не нужен при ручном кодировании, но критичен при вайбкодинге.

**Проблема:** ты не писал этот код. Ты не знаешь все детали. Как проверить что он правильный?

### Стратегия 1: Попроси AI объяснить

```
Объясни этот код как будто я не понимаю программирования.
Что делает каждая функция? Почему именно так?
```

Если AI не может объяснить просто — код скорее всего слишком сложный.

### Стратегия 2: Попроси AI сыграть адвоката дьявола

```
Ты только что написал этот код. Теперь притворись что ты 
злобный code reviewer. Найди всё что может быть неправильным.
```

### Стратегия 3: Чек-лист проверки

```
Проверь этот код по чек-листу:
□ Обрабатываются ли все ошибки?
□ Что будет если пользователь введёт пустую строку?
□ Есть ли SQL-инъекции или XSS?
□ Нет ли жёстко заданных паролей или ключей?
□ Понятны ли имена переменных и функций?
□ Есть ли дублирование кода?
```

### Стратегия 4: Тест как документация

Тесты показывают что код должен делать. Читая тесты — понимаешь намерение.

```
Напиши тесты для этого кода. Тесты должны служить документацией:
читая их, должно быть понятно что делает каждая функция.
```

## 5.3 Технический долг при вайбкодинге

**Технический долг** — это компромисс сейчас который придётся исправлять потом. Плохо названная переменная, скопированный код вместо функции, отсутствие обработки ошибок.

При вайбкодинге долг накапливается **в 10 раз быстрее** чем при ручном кодировании.

**Почему:**
- AI генерирует код быстро — и берёт «кредит» у будущего
- AI оставляет мусор: debug-логи, временные переменные, закомментированный код
- AI иногда решает одну проблему создавая три новых
- Без явного «убери мусор» → мусор остаётся

**Реальная история:** один разработчик после нескольких недель вайбкодинга обнаружил что его функция превратилась в трёхтысячестрочный монстр без модульных границ. Потратил три дня на рефакторинг.

### Как контролировать долг

**Рефакторинг по расписанию:**
```
# После завершения любой крупной фичи:
Посмотри на последние изменения. Найди:
- код который дублируется
- функции длиннее 30 строк
- непонятные имена переменных
- закомментированный код
- debug-логи
Почисти и упрости. Не ломая функциональность.
```

**Признаки что пора рефакторить:**
- Ты перестаёшь понимать что делает конкретная часть кода
- AI начинает делать странные решения («потому что так уже сложилось»)
- Одно изменение ломает что-то несвязанное
- Тесты занимают больше времени чем сама разработка

### Когда рефакторить

✅ Рефакторь когда:
- Закончил крупную фичу
- Что-то сломалось и не понятно почему
- Добавляешь третью похожую функцию (правило трёх)

❌ Не рефакторь когда:
- Горит дедлайн
- Добавляешь первую или вторую похожую вещь
- Код работает и его не трогают

## 5.4 Граница контроля: когда взять управление руками

Вайбкодинг не значит «отдай всё AI и иди пить кофе». Есть задачи где нужно вмешаться.

### Когда перестать доверять модели

**1. Ты перестал понимать что происходит**
Если ты не можешь объяснить что делает твой код — это проблема. Не AI-проблема, а твоя.

```
Перестань генерировать. Попроси объяснить.
Если не можешь понять объяснение — сделай шаг назад, упрости задачу.
```

**2. AI зациклился**
Модель делает одно и то же снова и снова. Разные подходы — та же ошибка.

```
Признак: «Попробую другой подход» → делает то же самое
Решение: Начни новый чат. Опиши задачу иначе. Или сделай маленький кусок руками.
```

**3. Изменение ломает несвязанные вещи**
Значит архитектура запуталась. AI не может решить архитектурную проблему генерацией кода.

```
Остановись. Нарисуй на бумаге как устроены части. 
Реши архитектуру в голове. Потом объясни AI.
```

**4. Критическая безопасность**
Авторизация, платежи, работа с персональными данными — проверяй каждую строку.

### Как не потерять понимание кода

**Читай diff перед каждым принятием:**
Что добавилось? Что удалилось? Почему?

**Проси объяснения:**
«Объясни что делает эта функция и почему именно так»

**Пиши tests-as-documentation:**
Тесты которые показывают намерение, не только факт

**Веди PROMPTS.md:**
Сохраняй важные архитектурные решения которые ты принял

## 5.5 Работа с большими кодовыми базами

Когда проект вырастает — стандартные подходы перестают работать.

### Введение AI в контекст существующего проекта

Новый чат — AI ничего не знает о проекте. Нужно ввести в контекст.

**Быстрый способ:**
```
Вот структура моего проекта:
[вставь tree или ls вывод]

Ключевые файлы:
- main.py: [что делает]
- database.py: [что делает]
- config.py: [что делает]

Я хочу добавить [задача]. С чего начать?
```

**Инструменты индексации:**
- `repomix` — упаковывает весь репозиторий в один текстовый файл
- `git-ingest` — аналог для больших репозиториев
- Claude Code — сам может читать файлы по мере необходимости

### Стратегии навигации

**Модульность как навигация:**
Хорошо структурированный проект проще передать AI в контекст.
```
src/
  auth/      ← всё про авторизацию
  data/      ← всё про базу данных
  ui/        ← всё что видит пользователь
```

**Документация как память:**
После каждой крупной сессии обновляй README или добавляй комментарии:
```python
# Эта функция делает X. 
# Важно: нельзя вызывать до инициализации Y.
# Добавлено [дата] для решения [проблема].
```

## 5.6 Будущее профессии: честный разговор

Куда движется рынок. Что автоматизируется. Что остаётся за человеком.

### Что автоматизируется точно

**Написание шаблонного кода**
CRUD операции, стандартные формы, типичные API — это уже автоматизировано. Это делает AI.

**Поиск и исправление простых багов**
Когда ошибка понятна и изолирована — AI исправит за секунды.

**Рефакторинг по правилам**
Переименование, извлечение функций, стандартизация — механическая работа AI.

**Написание тестов для существующего кода**
Покрытие тестами — AI делает хорошо.

**Документация**
Написать README, API-документацию, комментарии — AI справляется.

### Что остаётся за человеком

**Понимание проблемы пользователя**
AI не знает твоих пользователей. Ты знаешь.

**Архитектурные решения**
Выбор подходов, trade-offs, «что будет если мы вырастем в 10 раз» — суждение.

**Judgment: когда AI неправ**
Увидеть что правдоподобный ответ неправильный. Это требует экспертизы.

**Этика и ответственность**
AI не несёт ответственности. Ты несёшь.

**Нестандартные задачи**
Для всего что AI не видел в обучении — нужен человек.

## 5.7 Авторство и профессиональная идентичность

*Что значит быть специалистом когда AI делает большую часть работы?*

Это реальный вопрос который задают многие. Ответ на него определяет как ты строишь карьеру.

**Старый вопрос:** «Ты умеешь писать CSS?»
**Новый вопрос:** «Ты умеешь получить правильный CSS-результат с помощью AI?»

Это разные навыки. Но второй не менее ценный — он быстрее, мощнее, и позволяет делать больше.

**Где проходит граница ответственности:**
- Код написал AI → ты его принял → это твой код
- Бот ответил неправильно → твой бот → твоя ответственность
- Продукт сломался → твой продукт → твоя ответственность

Вайбкодинг не снимает ответственности. Он меняет инструмент, не роль.

## 5.8 Как строить карьеру в этом контексте

**Не конкурируй с AI в том что AI делает хорошо**
Не стоит специализироваться на написании шаблонного кода.

**Развивай то что AI не заменит**
- Глубокое понимание домена (ты знаешь дизайн, редактуру, маркетинг)
- Умение работать с людьми и понимать их проблемы
- Суждение: видеть когда AI неправ
- Архитектурное мышление

**Учись учиться быстро**
Инструменты меняются каждые месяцы. Способность быстро осваивать новое — ценнее знания конкретного инструмента.

**Комбинируй профессии**
Дизайнер который вайбкодит = дизайнер который может проверить что разработчик сделал правильно. Главред который вайбкодит = главред который может автоматизировать редакцию. Это другой уровень.

**Не жди стабилизации**
Через 6 месяцев AI будет лучше. Через год — ещё лучше. Те кто начал сейчас — будут на год впереди тех кто ждёт «пока устаканится».

## 5.9 Демо-день: как показывать продукт

**5 минут на человека:**

**Проблема (30 сек):**
«Раньше я тратил [сколько] на [что]. Вот как это выглядело.»

**Продукт (2 мин):**
Живое демо. Не слайды. Реальная демонстрация. Если что-то сломается — скажи «вот, AI иногда так делает» и продолжай.

**Результат (1 мин):**
«Теперь это занимает [сколько]. Я использую это [как часто]. Это уже помогло мне [конкретный пример].»

**Что узнал (30 сек):**
Один неожиданный инсайт. Что тебя удивило в процессе.

**Что дальше (30 сек):**
Следующий конкретный шаг.
