## 4.1 Контекстное окно глубже

В прошлой главе мы познакомились с контекстным окном. Теперь копнём глубже.

Из книги «Vibe Coding»:

> *«Мы явно сказали Claude Sonnet: «Никогда не запускай ps aux — это переполнит твоё контекстное окно». Сначала агент выполнял указание. Когда разговор заполнился на 50% контекстного окна — сначала попробовал ps с фильтром. Когда это не сработало — запустил запрещённую команду ps aux, обвалив сессию.»*

**Правило:** Чем больше у AI на «столе» — тем хуже он соображает. Исследования показали: большинство моделей начинают деградировать уже при 3000 токенах — намного раньше рекламируемого лимита.

**Что происходит при переполнении:**
- Генерирует менее связные ответы
- Забывает детали из 30 секунд назад
- Игнорирует явные инструкции (даже написанные ЗАГЛАВНЫМИ БУКВАМИ)
- Противоречит сам себе

**Как бороться:**
- Команда `/context` — смотришь процент заполнения
- При 70% → новый чат
- Разбивай большие задачи на маленькие в отдельных чатах

## 4.2 Скиллы: сохранённые промпты

**Скилл** — это промпт который сохранён в файле и вызывается одной командой через `/`.

**Аналогия:** Представь что у тебя есть кнопка на пульте. Нажал → включился сценарий. Не нужно каждый раз объяснять что делать.

**Где хранятся:**
```
.claude/
  skills/
    tone-of-voice/
      skill.md
    find-news/
      skill.md
    brief-generator/
      skill.md
```

**Формат skill.md:**
```markdown
---
name: brief-generator
description: Создаёт бриф на основе разговора с клиентом
---

Задай клиенту 7 вопросов по одному:
1. Что за проект?
2. Какова цель?
3. Кто аудитория?
4. Какой бюджет?
5. Какие сроки?
6. Есть ли референсы?
7. Что категорически нельзя?

После ответов — сформируй структурированный бриф.
Сохрани в briefs/[дата]-brief.md
```

**Вызов:** просто напиши `/brief-generator`

**Примеры скиллов по профессиям:**

| Профессия | Скилл | Что делает |
|-----------|-------|-----------|
| Главред | `/контент` | Анализ темы → структура → черновик в твоём стиле |
| Дизайнер | `/бриф` | Интервью с клиентом → готовый бриф |
| СМО | `/дайджест` | Собирает метрики → готовит еженедельный отчёт |
| Архитектор | `/смета` | Параметры объекта → расчёт стоимости |
| CRM маркетолог | `/письма` | Сегмент + оффер → варианты писем |
| Моушн дизайнер | `/сторибород` | Бриф → детальный сторибород сцен |
| UX researcher | `/инсайты` | Транскрипт → структурированные инсайты |

## 4.3 Агенты: параллельные помощники

**Агент** — это отдельный Claude который запускается со своей памятью и работает параллельно.

**Аналогия из книги «Vibe Coding»:**
> *«Представь задачу: купить квартиру, машину, найти стоматолога. Ты можешь делать это последовательно — или запустить трёх помощников параллельно. Каждый работает в своей «комнате» и не мешает другим. Каждый приносит результат. Ты получаешь всё сразу.»*

**Разница скилл vs агент:**

| | Скилл | Агент |
|--|-------|-------|
| Контекст | Текущий чат | Отдельное окно |
| Работа | Последовательно | Параллельно |
| Вызов | `/название` | Через промпт |
| Использование | Частые задачи | Параллельные исследования |

**Как запустить субагентов:**
```
Запусти 3 субагента параллельно:
1. Проанализируй мой продукт (prd.md) и найди сильные стороны
2. Найди слабые стороны и риски
3. Предложи 5 конкретных улучшений

Каждый агент сохраняет результат в отдельный файл в папке analysis/
```

**Когда использовать:**
- Параллельное исследование нескольких направлений
- Несколько перспектив на одну проблему
- Когда задача делится на независимые части

## 4.4 MCP: руки для Claude

**MCP (Model Context Protocol)** — протокол который позволяет Claude подключаться к внешним сервисам.

Из «The Vibe Coding Bible»:
> *«MCP — это как дать AI руки. Без MCP AI — умный советник без рук. С MCP — он может открыть Figma, прочитать Google Sheets, отправить письмо, создать задачу в Notion.»*

**MCP есть практически у любого крупного сервиса:**
- Figma, Notion, Google Sheets, Gmail
- Telegram, Slack, GitHub
- Любая база данных

**MCP под твою профессию:**

**🎬 Моушн дизайнер**
→ **Figma MCP**: Claude открывает макет → описывает фреймы → пишет анимационный сценарий → сторибород

**🔍 UX researcher**
→ **Notion MCP**: Claude читает транскрипт интервью → структурирует инсайты → сохраняет в базу знаний в Notion

**🎨 Дизайнеры**
→ **Figma MCP**: Claude видит дизайн → описывает правки → готовит комментарии для клиента

**✍️ Главреды**
→ **Notion MCP + Google Docs MCP**: Claude пишет прямо в твои документы

**📊 СМО**
→ **Google Sheets MCP + Gmail MCP**: читает метрики → анализирует → отправляет отчёт

**🏗 Архитектор**
→ **Google Sheets MCP**: параметры объекта → смета → записывает в таблицу

**📧 CRM маркетолог**
→ **Google Sheets MCP + Gmail MCP**: читает базу → сегментирует → пишет письма

**Установка:**
```bash
claude mcp add figma
claude mcp add gdrive
claude mcp add gsheets
claude mcp add gmail
```

## 4.5 MEMORY.md и CLAUDE.md: долгосрочная память

Из «The Vibe Coding Bible»:

> *«Новый сотрудник читает корпоративный регламент в первый день — культура, стандарты, неписаные правила. Context-файлы — это тот же регламент для твоего AI. Без них каждая сессия — снова первый день.»*

**Два важных файла:**

**`MEMORY.md`** — что Claude должен знать о тебе и проекте:
```markdown
# Кто я
Главред медиа о технологиях. Аудитория — менеджеры 30-45 лет.
Тон: умный, но без занудства. Никакого корпоративного языка.

# Мой продукт
Инструмент для контент-планирования с базой данных.

# Важные правила
- Всегда пиши по-русски
- Никаких маркдаун-заголовков в постах
- Абзацы не длиннее 3 предложений
```

**`CLAUDE.md`** — технические правила проекта (из Vibe Code Bible):
```markdown
# Project Context

## Stack
- Framework: Next.js
- Styling: Tailwind CSS
- Database: Supabase

## Code standards
- Комментарии только если код неочевиден
- Всегда обрабатывай состояния загрузки и ошибок

## What NOT to do
- Не устанавливай пакеты без разрешения
- Не меняй схему базы без согласования
```

**Зачем это нужно:**
- Не нужно каждый раз объяснять кто ты и что делаешь
- Claude следует твоим стандартам автоматически
- Файлы хранятся в репозитории — работают в любом инструменте

## 4.6 Команда агентов-дебатёров

Из книги «Vibe Coding»:

> *«Мы создали команду из трёх агентов: Генератор придумывает идеи, Адвокат защищает каждую, Критик атакует. В итоге получаешь не просто список идей — а идеи которые пережили спор и стали лучше.»*

Это экспериментальная функция Claude Code. Включается через:
```
Добавь в настройки Claude Code переменную:
CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

**Как это работает:**
1. Генератор читает твой PRD → предлагает идеи для улучшения
2. Адвокат берёт каждую идею → объясняет почему хорошая
3. Критик берёт каждую идею → объясняет почему плохая
4. Агенты разговаривают между собой
5. Ты получаешь проверенный результат с аргументами обеих сторон

## 4.7 Контент-пайплайн: система из скиллов

Для главредов и всех кто создаёт контент — это мощнейший инструмент.

**Система из четырёх скиллов:**

```
1. /tone-of-voice
   Читает твои тексты → создаёт профиль стиля
   (как ты пишешь, что любишь, что никогда не используешь)

2. /topic-map  
   Анализирует темы → строит карту
   (что растёт, что заброшено, что нельзя)

3. /find-news
   Открывает браузер → ходит по источникам → список новостей за сегодня
   фильтрует по твоим темам

4. /write-posts
   Берёт новости → запускает агентов-копирайтеров
   для каждой новости — пишет пост в твоём стиле
```

Одна команда `/write-posts` → готовые черновики постов на сегодня.

**Применимо не только для главредов:**
- СМО: `/weekly-report` — собирает данные → пишет отчёт
- Дизайнер: `/client-update` — берёт правки → готовит письмо клиенту
- CRM маркетолог: `/campaign-brief` — сегмент + оффер → готовый бриф

## 4.8 Модульность: почему важна структура

Из книги «Vibe Coding»:

> *«Исследования показали: разработчики которые работают в запутанных немодульных системах в 9 раз чаще уходят или теряют работу. И AI работает так же плохо в запутанном коде. Команда в OpenAI: у нас был хороший темп разработки нового проекта. Когда перенесли в монолит — темп упал катастрофически. Те же инженеры, те же инструменты.»*

**Для твоего продукта это означает:**
Просить Claude Code держать код структурированным:

```
Создай отдельные файлы:
- data.js — вся работа с базой данных
- ui.js — всё что показывается пользователю
- logic.js — бизнес-логика

Не смешивай их между собой.
```

Модульный код = Claude Code лучше с ним работает.
