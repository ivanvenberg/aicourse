## 3.1 Что такое агент

**Агент** — это AI-система которая не просто отвечает на вопрос, а выполняет многошаговые задачи используя инструменты.

**Чат-модель** (Claude в браузере):
```
Ты → запрос
  → Модель думает
  → Модель отвечает
```

**Агент** (Claude Code):
```
Ты → задача
  → Агент думает
  → Агент использует инструмент (читает файл, запускает код, ходит в браузер)
  → Видит результат инструмента
  → Думает снова
  → Использует другой инструмент
  → ... (много шагов)
  → Сообщает результат
```

Ключевое отличие: **цикл мышление → действие → наблюдение → мышление**.

## 3.2 Архитектуры агентов

### ReAct (Reasoning + Acting)
Самый популярный паттерн. Агент чередует рассуждение и действие.

```
Задача: «Найди ошибку в коде и почини»

Мысль: «Нужно сначала прочитать файл»
Действие: read_file("main.py")
Наблюдение: [содержимое файла]

Мысль: «Вижу ошибку в строке 47, нужно проверить как используется функция»
Действие: search_in_files("process_data")
Наблюдение: [использования функции]

Мысль: «Проблема в том что функция ожидает список, но получает строку»
Действие: edit_file("main.py", ...)
Наблюдение: «Файл изменён»

Мысль: «Нужно проверить что исправление работает»
Действие: run_tests()
Наблюдение: «Все тесты прошли»

Ответ: «Исправил ошибку в строке 47...»
```

**Почему ReAct работает:** рассуждение видно в контексте, каждое действие основано на предыдущем наблюдении. Модель не «угадывает» — она реагирует на реальные результаты.

---

### Plan-and-Execute
Сначала планируем всё целиком, потом выполняем по плану.

```
Фаза 1 — Планирование:
«Создать контент-пайплайн»
→ План: [1. скилл tone-of-voice, 2. скилл topic-map, 3. скилл find-news, 4. агент-копирайтер]

Фаза 2 — Выполнение:
→ Создаём по плану шаг за шагом
```

**Когда лучше ReAct:** задачи где не знаешь заранее что понадобится.
**Когда лучше Plan-and-Execute:** задачи с понятной структурой, большие проекты.

---

### Reflexion (Рефлексия)
Агент после выполнения задачи критикует свой результат и улучшает его.

```
Шаг 1: Пишет код
Шаг 2: Запускает код
Шаг 3: Видит что тест не прошёл
Шаг 4: Анализирует почему
Шаг 5: Исправляет
Шаг 6: Повторяет пока все тесты не пройдут
```

Claude Code делает это автоматически когда запускает тесты.

## 3.3 Инструменты агентов и их ограничения

Агент без инструментов — советник без рук. Инструменты — это что агент может делать.

### Браузер
**Что умеет:** открывать сайты, кликать кнопки, читать текст, заполнять формы.
**Как может сломаться:**
- Сайт требует авторизацию
- Защита от ботов (CAPTCHA)
- Динамический контент который загружается через JavaScript
- Rate limiting

**Практика:** для надёжного чтения — использовать официальные API сайта если есть.

### Терминал / Bash
**Что умеет:** запускать программы, читать файлы, устанавливать пакеты, делать запросы.
**Как может сломаться:**
- Команда требует sudo (права администратора)
- Зависимость не установлена
- Разные команды на Mac и Windows
- Бесконечный цикл

**Практика:** всегда проверяй что установлено перед запуском. Используй `which python3` перед `python3 script.py`.

### Файловая система
**Что умеет:** читать, создавать, изменять, удалять файлы.
**Как может сломаться (и сломалось у реальных людей):**
- Удалил файлы без запроса «ты уверен?»
- Перезаписал важный файл
- Удалил тестовые файлы которые казались ненужными

**Практика:** `git commit` перед каждой большой операцией. Это точка сохранения.

### API
**Что умеет:** отправлять запросы к внешним сервисам, читать данные.
**Как может сломаться:**
- Rate limiting (слишком много запросов)
- Изменился формат ответа API
- Кончились деньги/лимиты
- Нет обработки ошибок

### Code Interpreter
**Что умеет:** запускать Python, анализировать данные, строить графики.
**Как может сломаться:**
- Код работает в sandbox без доступа к интернету
- Большие файлы не влезают
- Библиотека не установлена

## 3.4 Мульти-агентные системы

Один агент делает одну задачу. Несколько агентов могут работать параллельно.

**Почему это мощно:**
- Параллельность: три задачи одновременно вместо последовательно
- Специализация: каждый агент настроен на свою работу
- Проверка: один агент пишет, другой проверяет

**Команда дебатёров:**
```
Генератор → предлагает идеи
Адвокат → защищает каждую идею
Критик → атакует каждую идею
→ консенсус который пережил проверку
```

**Архитектура оркестрации:**
```
Оркестратор (главный агент)
├── Субагент 1: анализирует данные
├── Субагент 2: пишет код
└── Субагент 3: проверяет результат
```

Оркестратор координирует, субагенты выполняют. Каждый субагент — своё контекстное окно, своя задача.

**Проблема контентии (agent contention):**
Когда несколько агентов пытаются изменить один файл одновременно — конфликты. Решение: каждый агент работает в отдельном файле/модуле.

## 3.5 Декомпозиция сложных задач

**Почему важно:**
Если дать агенту слишком большую задачу — он начнёт делать ошибки. Теряет суть, срезает углы, «забывает» требования.

**Граф задач:**
```
Создать контент-пайплайн
├── Скилл: tone-of-voice
│   ├── Прочитать 10 постов
│   ├── Выделить паттерны стиля
│   └── Сохранить профиль
├── Скилл: topic-map
│   ├── Проанализировать темы
│   └── Построить карту
└── Агент: copywriter
    ├── Читать: tone-of-voice
    ├── Читать: topic-map
    ├── Получить новость
    └── Написать пост
```

**Правило Least-to-Most в агентах:**
Начинай с минимально работающей версии. Добавляй по одной фиче, каждый раз проверяя.

**Передача контекста между сессиями:**
```
# В конце каждой сессии:
Создай файл session-summary.md с:
- что сделано
- какие решения приняты
- что осталось
- любые важные детали которые нужны в следующей сессии
```

## 3.6 Контекстные файлы: долгосрочная память

У AI нет памяти между сессиями. Контекстные файлы — это способ дать ему «память».

**CLAUDE.md** — правила проекта (читается Claude Code автоматически):
```markdown
# Project Context

## Стек
- Frontend: HTML + CSS + JS (без фреймворков)
- База данных: Supabase
- Деплой: Vercel

## Правила
- Комментарии только если код неочевиден
- Всегда обрабатывай состояния загрузки и ошибок
- Никогда не удаляй файлы — перемещай в /archive/

## Что НЕ делать
- Не устанавливай пакеты без разрешения
- Не меняй схему базы данных без согласования
```

**MEMORY.md** — информация о тебе и продукте:
```markdown
# Кто я
[Профессия], работаю в [индустрия].

# Мой продукт
[Одно предложение].

# Важные правила
- [Стиль, тон, предпочтения]
```

**AGENTS.md** — разрешения для агентов:
```markdown
## Разрешения
- MAY: читать и редактировать файлы в /src/
- MAY: запускать npm test
- MAY NOT: коммитить напрямую в main
- MAY NOT: изменять .env файлы

## При неопределённости
→ Спрашивай перед действием, не после
```

## 3.7 Три проблемы агентов которые тебя удивят

### «Проблема с детьми»
Ты: «Войди в горящий дом и спаси семь детей»
Агент: «Выполнено! Принёс пятерых и отключил двух. Проблема решена.»

Агент делает видимость выполнения задачи. Вместо починки тестов — отключает их. Вместо решения проблемы — убирает проверку которая её обнаруживает.

**Защита:**
```
После выполнения задачи — явно проверь:
- сколько тестов было до?
- сколько тестов сейчас?
- все ли тесты реально тестируют что должны?
```

### «Картонные кексы»
Агент сделал 9 кексов. Пять настоящих, четыре — из картона. Выглядят одинаково, пока не укусишь.

Агент хардкодит значения чтобы тесты «проходили». Создаёт видимость работающего кода.

**Защита:** проверяй не только что тесты зелёные, но что они реально тестируют что-то.

### «Слизняк и мусорщик»
После сессии агент оставляет:
- Debug `console.log` везде
- Переменные `interim_result5`, `backup_just_in_case`
- Закомментированный старый код
- Временные файлы

```
После завершения задачи:
Почисти код — удали все отладочные логи, 
неиспользуемые переменные, закомментированный код.
```

## 3.8 Скиллы: переиспользуемые промпты

**Скилл** — сохранённый промпт который вызывается одной командой через `/`.

**Место хранения:**
```
.claude/
  skills/
    [название]/
      skill.md
```

**Формат skill.md:**
```markdown
---
name: brief-generator
description: Создаёт структурированный бриф после интервью с клиентом
---

Задай клиенту 7 вопросов по одному. Жди ответа перед следующим вопросом.
[список вопросов]

После всех ответов — создай структурированный бриф в формате:
[формат]
Сохрани в briefs/[дата]-[название].md
```

**Примеры скиллов по профессиям:**

| Профессия | Скилл | Что делает |
|-----------|-------|-----------|
| Главред | `/контент` | Тема → структура → черновик в твоём стиле |
| Дизайнер | `/бриф` | Интервью с клиентом → структурированный бриф |
| СМО | `/отчёт` | Данные → еженедельный отчёт |
| Архитектор | `/смета` | Параметры → расчёт стоимости |
| CRM маркетолог | `/письма` | Сегмент + оффер → варианты писем |
| Моушн | `/сторибород` | Бриф → детальный сторибород |
| UX researcher | `/инсайты` | Транскрипт → структурированные инсайты |

## 3.9 MCP: инструменты для агентов

**MCP (Model Context Protocol)** — стандарт который позволяет AI подключаться к внешним сервисам.

Без MCP: AI читает и пишет файлы на твоём компьютере.
С MCP: AI открывает Figma, читает Google Sheets, создаёт задачи в Notion.

```bash
claude mcp add figma    # дизайнеры, моушн
claude mcp add gdrive   # главреды
claude mcp add gsheets  # СМО, CRM, архитектор
claude mcp add gmail    # СМО, CRM маркетолог
claude mcp add notion   # UX researcher, главреды
```

После установки — AI видит твои реальные файлы и данные.
