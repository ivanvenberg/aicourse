## 2.1 Промпт-инженерия как дисциплина

Промпт-инженерия — не «магические фразы». Это понимание того как модель интерпретирует входной текст.

**Ключевой принцип:** модель достраивает то что ты дал. Дал вопрос → достраивает ответ. Дал вопрос с примером → достраивает похожий ответ. Дал рассуждение → продолжает рассуждение.

---

## 2.2 Zero-shot, few-shot, chain-of-thought

### Zero-shot
Просто описываешь задачу. Никаких примеров.

```
Классифицируй тональность:
«Сервис работает нормально, ничего особенного»
```

**Когда работает:** задача простая, модель видела много похожего.
**Когда не работает:** нестандартный формат, специфическая логика.

---

### Few-shot
Даёшь 2–5 примеров вход→выход перед реальным запросом.

```
Формат: [тема] → [заголовок]

пример: маркетинг в кризис → «Как не потерять клиентов когда всё горит»
пример: дизайн без бюджета → «5 сервисов которые заменят дизайнера»
пример: работа с текстом → ?
```

**Почему работает:** модель завершает «последовательность» основываясь на паттерне. Это буквально «следующий токен должен выглядеть как предыдущие ответы».

**Когда использовать:** нужен конкретный формат вывода, нестандартная классификация, нужна консистентность.

---

### Chain-of-Thought (CoT)
Просишь модель рассуждать вслух перед ответом.

```
❌ «Какой подход выбрать для хранения данных?»

✅ «Мне нужно хранить данные. Рассмотри плюсы и минусы каждого варианта:
   localStorage, Supabase, Firebase. Потом дай рекомендацию.»
```

**Почему работает:** промежуточные шаги попадают в контекст и влияют на следующие токены. Модель рассуждает с помощью своего собственного вывода.

**Критично при:**
- Многошаговых рассуждениях
- Архитектурных решениях
- Отладке («объясни что делает каждая строка»)

## 2.3 Системные промпты

**Системный промпт** — инструкция которая идёт перед разговором. Читается при каждом запросе.

```
[Системный промпт] → задаётся один раз, читается всегда
[История разговора] → накапливается
[Запрос пользователя]
```

**Что класть:**
- Роль и контекст
- Постоянные ограничения
- Стиль ответов
- Правила которые нельзя нарушать

**Для тебя это CLAUDE.md и MEMORY.md** — по сути системный промпт для проекта. Модель читает их при каждом старте.

```markdown
# MEMORY.md — системный промпт для Claude Code
## Кто я
Главред медиа о технологиях. Аудитория — менеджеры 30-45 лет.
## Правила
- Всегда пиши по-русски
- Абзацы не длиннее 3 предложений
```

## 2.4 Почему LLM не умеет считать

Это самый важный практический инсайт главы.

Модель генерирует токены один за другим. Когда она «считает» — она не вычисляет математически. Она предсказывает какие цифры правдоподобно стоят после твоего вопроса.

**Что это значит:** один и тот же вопрос с одними и теми же данными даст разные числа у разных людей. AI галлюцинирует цифры.

Python-скрипт детерминирован. Один код + одни данные = один результат всегда.

**Правило:** хочешь точные числа → проси написать скрипт → запускай скрипт.

## 2.5 Управление контекстом когда проект растёт

Три стратегии:

**Стратегия 1: Сжатие**
```
Суммаризируй нашу сессию: что сделали, какие решения приняли,
что осталось. Сохрани в context-summary.md
```

**Стратегия 2: Модульные чаты**
Каждая задача — отдельный чат. Между ними — файлы.
```
Чат 1 → создаёт schema.md
Чат 2 → читает schema.md → создаёт api.md
Чат 3 → читает api.md → создаёт UI
```

**Стратегия 3: CLAUDE.md**
Критические правила → в CLAUDE.md. Читается при каждом запуске.

## 2.6 RAG: база знаний вместо памяти модели

**RAG (Retrieval-Augmented Generation)** — AI ищет в твоих данных и отвечает на основе найденного.

**Без RAG:**
«Какова политика возврата?» → AI придумывает правдоподобное

**С RAG:**
«Какова политика возврата?» → система ищет в базе → находит нужный фрагмент → AI отвечает точно по документу

**Как устроено:**
1. Документы разбиваются на кусочки (чанки)
2. Каждый чанк превращается в числовой вектор (эмбеддинг)
3. При запросе — ищем похожие векторы → находим нужные чанки
4. Вставляем чанки в контекст → AI отвечает по ним

Подробнее — в главе 4, когда будем строить бота.

## 2.7 База данных: почему нужна

**localStorage** (браузер): данные хранятся у одного пользователя в одном браузере. Закрыл → всё там. Другой браузер → пусто.

**База данных** (облако): данные хранятся на сервере. Любое устройство, любой пользователь с доступом, история сохраняется.

**Как устроена:**
Таблица = Excel-лист. Строка = одна запись. Колонка = одно поле.

```
tasks:
| id | title           | status | created_at |
|----|-----------------|--------|------------|
| 1  | Написать бриф  | done   | 2025-05-10 |
| 2  | Позвонить клиенту | todo | 2025-05-11 |
```

AI пишет весь код для работы с базой. Тебе нужно только объяснить что хранить.

## 2.8 Supabase

Supabase — PostgreSQL (серьёзная база) с удобным интерфейсом и бесплатным тарифом.

**Что получаешь:**
- База данных в облаке
- Интерфейс где видишь таблицы
- Автоматический API
- Аутентификация пользователей
